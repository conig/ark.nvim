# Ark v0.1.0-alpha.1 (planned and unpublished)

Status: this release has not yet been tagged or published. Before cutting the
tag, reconcile the `Unreleased` changelog into these notes and perform the
manual release checklist from the exact tag commit:

```sh
just verify-product
just verify-upstream-compat
just verify
just benchmark
./scripts/package-release.sh
```

Review the retained test and benchmark logs, verify the generated SHA-256 file
and `build-metadata.json`, and clean-install that exact package before creating
the tag or publishing assets.

This is planned as the first Ark.nvim product release. It packages the detached
`ark-lsp` server for the supported Linux x86_64 platform and establishes the
versioned plugin/LSP/bridge compatibility contract.

## Highlights

- Installs an optimized, checksummed `ark-lsp` without requiring Rust or Cargo.
- Keeps versioned installs under Neovim's data directory and preserves the
  previous working release for `:Ark rollback`.
- Reports build target, commit, Rust compiler, optimization profile, and bridge
  protocol metadata through `ark-lsp --version --json`.
- Adds product-owned local Rust, Neovim, bridge-package, clean-install, and live
  session verification gates.

## Supported environment

- Linux x86_64 with glibc 2.35 or newer
- Neovim 0.11.3 or newer
- R 4.2 or newer
- tmux as the canonical backend; the built-in terminal backend remains an
  additive, narrower alternative

Windows managed sessions and prebuilt macOS artifacts are not part of this
release tier.

## Install and rollback

Pin the plugin to `v0.1.0-alpha.1`, then run `:Ark install` after it has loaded
(the recommended plugin-manager build hook does this automatically). Ark
downloads the release asset and checksum, verifies the checksum plus embedded
version, target, optimized profile, and bridge schema, then atomically changes
the current-version pointer.

Rollback must keep all product layers aligned: pin and load the previous plugin
tag first. Its build hook normally activates the matching binary; otherwise run
`:Ark rollback` from that previous checkout. Ark refuses to activate an
incompatible previous binary. An interrupted or invalid upgrade leaves the
current version unchanged, and an orphaned installer lock is safely reclaimed.
